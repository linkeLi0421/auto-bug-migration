# Patch Merge Utility Overview

`script/patch_merge.py` is a helper that:

1. Loads patch dictionaries previously generated by `revert_patch_test`.
2. Builds a compatibility graph where each node corresponds to a patch set for a specific bug.
3. Detects fully compatible groups (cliques) of bugs whose patches can coexist.
4. Optionally visualizes the graph in Graphviz DOT format.
5. Tracks “local” bugs (specified in `local_bug_compatibility`) and displays them as red nodes in the exported graph.
6. Records and optionally regenerates patches for bugs whose patch commit differs from the shared triggering commit they have with another bug in the same group.

## Basic Usage

First, source the environment setup:

```bash
source script/setenv.sh
```

Then run the script:

```bash
OPENAI_API_KEY=dummy python3 script/patch_merge.py \
  --bug_distribution_csv $LOG_PATH/c-blosc2.csv \
  script/patches.pkl.gz \
  --graphviz_output log/patch_compatibility.dot
```

Key positional/optional arguments:

| Argument | Purpose |
| --- | --- |
| `cache_file` | Path to the pickled patch dictionary (`*.pkl` or `*.pkl.gz`). |
| `--bug_distribution_csv` | CSV describing bug trigger status per commit (same format as `prepare_transplant`). Required for commit-aware compatibility checks. |
| `--graphviz_output` | Optional DOT file path for visualizing the compatibility graph. |

The script logs:

- Graph stats (`nodes`, `edges`, number of “local” nodes).
- Detected fully compatible groups (cliques).
- Pending patch refreshes that affect the first group (Group 1), indicating any mismatched patch commits.

## Automatic Patch Refresh (Optional)

To have `patch_merge` call `revert_patch_test` when a Group 1 bug needs a refreshed patch, provide all the `--revert_*` flags:

```bash
OPENAI_API_KEY=dummy python3 script/patch_merge.py \
  --bug_distribution_csv $LOG_PATH/c-blosc2.csv \
  script/patches.pkl.gz \
  --graphviz_output log/patch_compatibility.dot \
  --revert_target_test_result $LOG_PATH/c-blosc2.csv \
  --revert_bug_info $BUGINFO_PATH \
  --revert_build_csv $LOG_PATH/c-blosc2_builds.csv \
  --revert_target c-blosc2 \
  --revert_output_dir log/revert_patch
```

These flags map 1:1 to the `revert_patch_test.py` arguments:

| Flag | Forwarded argument |
| --- | --- |
| `--revert_target_test_result` | `target_test_result` CSV |
| `--revert_bug_info` | `--bug_info` JSON |
| `--revert_build_csv` | `--build_csv` |
| `--revert_target` | `--target` |
| `--revert_output_dir` | Optional directory where each `revert_patch_test` run’s stdout/stderr is captured |

**Important:** The environment variables required by `revert_patch_test` (e.g., `REPO_PATH`, `TESTCASES`) still must be set before running `patch_merge`. The script simply reuses the same helper but does not manage those environment requirements.

When these flags are supplied, `patch_merge`:

1. Records pending refresh requests during compatibility analysis (commit mismatch).
2. Prints “Pending patch refreshes impacting Group 1” after listing the groups.
3. Triggers `revert_patch_test` per entry at that point (but only once per bug+commit).
4. Logs whether each revert attempt succeeded or failed; actual refreshed patches must later be reloaded into the cache file if you want them reflected in future runs.

## Local Bug Nodes

`local_bug_compatibility` specifies local-only bugs associated with a remote bug’s patch. For each entry:

- A red node is added to the graph.
- Edges connect the local node to every patch node for the corresponding bug.
- Local nodes form a fully connected subgraph (clique) so they appear compatible with one another.

This provides a visual hint showing which local bugs are covered by a given remote patch set.

## Log Interpretation

Typical output segments:

- `Constructed compatibility graph: …` summarises patch nodes and compatibility edges.
- `Detected N fully compatible bug groups …` enumerates cliques ordered by size.
- `Pending patch refreshes impacting Group 1 …` lists only the bugs in Group 1 whose patch commit differs from the shared trigger commit. Once all relevant patches are regenerated and the cache file is updated, the section disappears.

If run with revert support, additional logs detail the `revert_patch_test` invocations (see `log/revert_patch/*.log`).

## Troubleshooting

- **Missing cache file**: Ensure the positional `cache_file` exists (e.g., `script/patches.pkl.gz`). If not, run `revert_patch_test` first to generate patches.
- **Permission errors during automatic refresh**: `revert_patch_test` performs git operations inside `oss-fuzz`. Run `patch_merge` with sufficient privileges (or pre-fix file ownership).
- **Environment variables**: `revert_patch_test` relies on `script/setenv.sh` variables (e.g., `REPO_PATH`, `TESTCASES`). Source that script or export the variables prior to running `patch_merge` when using auto-refresh mode.
