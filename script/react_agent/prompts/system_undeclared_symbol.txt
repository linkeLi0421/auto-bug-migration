Undeclared symbol/type/function errors (C/C++):
- Applies to: `use of undeclared identifier 'X'`, `unknown type name 'X'`, `call to undeclared function 'X'`, `implicit declaration of function 'X'`.

FIRST CLASSIFY THE DIAGNOSTIC:
- `use of undeclared identifier 'X'` => identifier path.
- `unknown type name 'X'` => type path.
- `call to undeclared function 'X'` or `implicit declaration of function 'X'` => function-call path.
- Do NOT mix these paths.
- In particular: KB/search results for a V1 function (`FUNCTION_DECL` / `FUNCTION_DEFI`) are NOT scope evidence for variable handling, and they do NOT justify `make_extra_patch_override` for an ordinary non-`__revert_*` undeclared function call.

Decision precedence (highest first):
1) Undeclared `__revert_*` function call => `make_extra_patch_override` (forward declaration/prototype handling).
2) Undeclared non-`__revert_*` function call => rewrite the caller; do NOT revive the function via `_extra_*`.
3) Undeclared identifier `X` => run KB scope check first, then choose file-scope insertion vs local declaration.
4) Unknown type name `X` / missing macro-like symbol => `make_extra_patch_override` first.

1) Undeclared function calls starting with `__revert_`:
   - `__revert_*` wrappers are expected. Do NOT remove/replace calls to them.
   - ALWAYS use `make_extra_patch_override` to add the forward declaration, even if `editable_hunk` is present.
   - Do NOT use `revise_patch_hunk` to revert the call back to the original function.
   - Workflow:
     a) Derive `func` by stripping the `__revert_<commit>_` prefix.
     b) `search_definition(symbol_name="func", version="v1")` to confirm the signature.
     c) If the active patch_key is NOT `_extra_*`: call `make_extra_patch_override(symbol_name="__revert_<commit>_<func>")`.
        Do NOT set `prefer_definition` here — only a prototype is needed.
        If the active patch_key IS `_extra_*`: insert the prototype into the hunk via `make_error_patch_override`.
   - Sanity check: the inserted code must be a prototype ending with `;`, not a macro or statement.

2) Undeclared function calls NOT starting with `__revert_`:
   - Treat the symbol as a function, not a variable.
   - These calls should be removed/adapted at the caller. Do NOT use `make_extra_patch_override`.
   - Do NOT insert a prototype, extern declaration, or full function definition.
   - If `editable_hunk` is present, use `revise_patch_hunk` to remove/adapt the call in `+` lines.
   - Otherwise use `make_error_patch_override` to rewrite the caller slice and remove/adapt the call.
   - You may use `search_definition(symbol_name="X", version="v1")` only to understand what the old helper did.
     A V1 `FUNCTION_DECL` / `FUNCTION_DEFI` result means only that `X` existed in V1.
     It does NOT authorize `make_extra_patch_override(symbol_name="X", prefer_definition=true)`.
   - Example: if `search_definition("node_cleanup_walker", "v1")` returns `FUNCTION_DEFI`, you still MUST rewrite the caller and MUST NOT call `make_extra_patch_override(symbol_name="node_cleanup_walker", prefer_definition=true)`.

3) Undeclared identifiers (for example `use of undeclared identifier 'g_ctx'`):
   - **KB-FIRST SCOPE CHECK (required before local-fix decision)**.
   - Before choosing a local declaration rewrite, check KB evidence for `X` in V1.
   - **RECLASSIFY when KB reveals a function**: if `search_definition` returns `FUNCTION_DECL` or `FUNCTION_DEFI` for `X`, the symbol is a function, not a variable.
     Reclassify to path 2 (undeclared non-`__revert_*` function call) and rewrite/remove the call site. Do NOT use `make_extra_patch_override` to insert a declaration or definition.
   - For non-function identifiers, KB/search is used to infer scope.
   - If KB shows `X` belongs to translation-unit / non-local scope (for example global/static declaration, related `VAR_DECL`, or analysis evidence across function boundaries), prefer `make_extra_patch_override(symbol_name='X')`.
   - Example: for `g_ctx`, if V1 KB/analysis shows it as a non-local variable, use `make_extra_patch_override` first.
   - Counter-example: for `node_cleanup_walker`, if KB returns `FUNCTION_DEFI`, reclassify to path 2 and remove/adapt the call — do NOT insert a prototype or definition.
   - **LOCAL-VAR FALLBACK (LLVMFuzzerTestOneInput)**.
   - Decision checklist (all must be true):
     a) Diagnostic is exactly `use of undeclared identifier 'X'`.
     b) Error location/function is inside `LLVMFuzzerTestOneInput`.
     c) `get_error_patch_context` returned non-empty `editable_hunk` (mixed `+`/`-`).
     d) KB scope check did NOT establish that `X` is non-local/global.
     e) `X` looks function-local (assigned/read/passed in function body) and is NOT a function call target, type name, macro token, or known global/extern symbol.
   - If all checklist items are true: use `revise_patch_hunk` and add a minimal local declaration in `+` lines near first use.
   - If KB indicates non-local/global scope, do NOT force a local declaration via `revise_patch_hunk`; use `make_extra_patch_override(symbol_name='X')` first.
   - If the checklist is not fully satisfied, fall back to generic identifier/type priority and use `make_extra_patch_override` before broader function-body rewrites.
   - Do not replace/drop the identifier inside the active function as a first step.

4) Unknown type name / missing macro-like symbol:
   - Prefer `make_extra_patch_override(symbol_name='X')` first.
   - Do not treat `unknown type name` as a local-variable declaration problem.

CRITICAL: when the active patch_key is `_extra_*`:
- Never call `make_extra_patch_override` while fixing an `_extra_*` hunk (it would extend the same hunk and can cascade).
- Fix the `_extra_*` hunk with `make_error_patch_override`.
- Keep correct `__revert_*` forward declarations; add their missing types instead of deleting them.
- For `unknown type name` errors (e.g. `size_t`, `ssize_t`) in `_extra_*` hunks:
  Use LINE-RANGE mode to **prepend** the include at the very beginning of the hunk:
    replace_start_line: 0
    replace_end_line: 0
    new_code_replace: "#include <stddef.h>"
  This inserts the include BEFORE all forward declarations without copying any code.
