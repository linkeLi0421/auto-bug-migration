Function signature change errors (too few/many arguments):
- These errors occur when a callee function's signature changed between V1 and V2.
- Error patterns: "too few arguments to function call", "too many arguments to function call"

CRITICAL - FUNCTION NAME PRESERVATION:
- The `error_func_code` / `focus_error_func_snippet` from get_error_patch_context shows the EXACT caller function name and signature.
- If the BASE slice has `__revert_5f664a_xmlParseTryOrFinish(...)`, your new_func_code MUST use that EXACT name.
- NEVER strip the `__revert_*` prefix! Writing `xmlParseTryOrFinish` instead of `__revert_5f664a_xmlParseTryOrFinish` causes redefinition errors.

FIX WORKFLOW - Follow these steps in order:

1. GET CALLER CODE: Use `get_error_patch_context` with the error file_path and line_number.
   - This returns `error_func_code` containing the caller function that has the bad call site.
   - The function signature in `error_func_code` is the EXACT signature you must preserve.
   - IMPORTANT: Check if the response includes a non-empty `editable_hunk` field — this changes step 3.

2. GET CALLEE SIGNATURE: Use `search_definition(callee_name, version="v2")` to find the NEW callee signature.
   - The callee name is the function being called at the error location.
   - V2 is the target version - this gives you the new expected signature.

3. FIX THE CALLER:
   A) If `editable_hunk` is non-empty (mixed hunk with both V1 and V2 code):
      Use `revise_patch_hunk` instead of `make_error_patch_override`.
      - Copy the `editable_hunk` text exactly.
      - Edit ONLY the `+` lines (V1 code being added) to fix the call site arguments.
      - Keep ALL `-` lines (V2 code) and context lines (no prefix) EXACTLY as they are.
      - Pass the edited text as `revised_hunk` to `revise_patch_hunk`.
   B) If `editable_hunk` is empty or absent (pure __revert_* function hunk):
      Use `make_error_patch_override` with a fixed version of the caller function.
      - Start with the EXACT function signature from `error_func_code` (including any __revert_* prefix).
      - Keep the ENTIRE function body from `error_func_code`.
      - Only modify the specific call site(s) to match the new callee signature.

RULES FOR FIXING CALL SITES:
- If parameters were ADDED to callee: use existing local variables if available, otherwise NULL/0
- If parameters were REMOVED from callee: delete them from the call
- Do NOT invent new struct fields or variables that don't exist in the caller
- Check the caller function for existing local variables that match the new parameter types
- When in doubt, use NULL for pointers and 0 for integers

EXAMPLE A (editable_hunk present — mixed hunk):
Error: "too many arguments to function call, expected single argument 'in', have 3 arguments"
get_error_patch_context returns editable_hunk:
  +            view_vcf(data, size, "w");     // VCF
  +            view_vcf(data, size, "wb");    // BCF
  -            view_vcf(ht_file);
search_definition("view_vcf", "v2") -> void view_vcf(htsFile *in)
Fix: edit only the '+' lines to match V2 signature, keep '-' lines unchanged:
  revised_hunk:
  +            view_vcf(ht_file);             // VCF
  +            view_vcf(ht_file);             // BCF
  -            view_vcf(ht_file);
Then call revise_patch_hunk(revised_hunk=...).

Error: "too few arguments to function call 'xmlParseEndTag2'"
error_func_code starts with: "__revert_5f664a_xmlParseTryOrFinish(xmlParserCtxtPtr ctxt, int terminate) {"

1. get_error_patch_context -> returns caller function with old call: xmlParseEndTag2(ctxt, tag)

3. make_error_patch_override -> new_func_code MUST start with:

   __revert_5f664a_xmlParseTryOrFinish(xmlParserCtxtPtr ctxt, int terminate) {
       // ... fixed body with updated call site ...
   }
