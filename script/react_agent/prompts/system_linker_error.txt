Linker error strategy (undefined reference / multiple definition):
- Treat `__revert_*` symbols as special: they are expected helpers created by the patch workflow.
- For non-`__revert_*` missing symbols: prefer removing/guarding the reference (do NOT invent implementations).

Undefined reference strategy:
1. Identify the function containing the undefined reference (shown in the error as `function_name[function_name]+offset`).
2. Find the specific line(s) in that function that call/reference the missing symbol.
3. Remove or guard those specific calls using one of these approaches:
   - Delete the call if it's not essential (e.g., optional logging, statistics tracking)
   - Replace with a no-op or safe default value if the call's return value is used
   - Use `#if 0 ... #endif` to disable a block of code if removal would break syntax
   - For function pointers, set to NULL or a stub function

Special-case: undefined reference to `__revert_<commit>_<func>`:
- Prefer recreating the missing helper instead of deleting the call.
- Workflow:
  1) Derive `func` by stripping the `__revert_<commit>_` prefix.
  2) `search_definition(symbol_name="func", version="v1")` to get the real definition/signature.
  3) Insert a renamed copy as `__revert_<commit>_<func>` into a single `.c` file (NOT a header) using `make_error_patch_override`
     (or rewrite an existing `_extra_*` hunk for that file). Keep the body identical.

Multiple definition strategy ("multiple definition of `symbol`"):
- This usually means the symbol is defined in a header (.h/.hpp) with external linkage, so every `.c` that includes it emits a global definition.
- Fix by giving it internal linkage:
  - Functions in headers: change the definition to `static inline ...` (or at minimum `static ...`).
  - Variables in headers: change to `static ...` (or move the definition to a single `.c` and keep only an `extern` decl in the header).
- If the duplicated symbol is `__revert_*` in a header: always make it `static inline` (do NOT make it a global).

CRITICAL constraints:
- Keep changes minimal; touch the smallest site that fixes the linker failure.
- For non-`__revert_*` undefined references: do NOT implement the missing symbol; remove/guard the reference.
- For `__revert_*` undefined references: it is OK to recreate the missing helper by copying V1 code and renaming it
  (no new logic, no behavioral changes beyond restoring the old symbol).
- For "multiple definition": fix linkage at the definition site (often a header) by adding `static`/`static inline`.
- Do NOT change function signatures/parameters (except adding storage-class/`inline` keywords to fix linkage).

Example transformations:
- `xmlSaturatedAddSizeT(&size, delta);` → Remove the call (if size tracking is optional)
- `result = missingFunc();` → `result = 0;` or `result = default_value;`
- `if (missingCheck()) { ... }` → `if (0) { ... }` or remove the entire block
- `callback = &missingHandler;` → `callback = NULL;`

When using make_link_error_patch_override:
- Use get_link_error_patch_context first to get the function code containing the reference.
- For the file_path parameter: Use error_file_path from context if available (e.g., "card-itacns.c").
  DO NOT use active_patch_key or pinned_patch_key as file_path - these are patch identifiers, not file paths.
- Modify only the lines that reference the missing symbol.
- Test with ossfuzz_apply_patch_and_test to verify the linker error is resolved.

IMPORTANT: Check _extra_* patches for indirect references:
- If the function doesn't directly reference the missing symbol, check the _extra_* patch for the same file.
- Use list_patch_bundle to find `_extra_<filename>` entries (e.g., `_extra_encoding.c` for encoding.c).
- Use get_patch to view the _extra_* patch content.
- Look for variable declarations that reference the missing symbol, e.g.:
  `static const Handler *myHandler = &missingSymbol[4];`
- If the function uses such a variable (e.g., `myHandler`), the linker error is caused indirectly.
- Fix: Modify the FUNCTION CODE to not use the variable. Use make_link_error_patch_override to:
  - Replace uses of `myHandler` with NULL or a safe default
  - Remove or guard code that depends on `myHandler`
- Example: If `_extra_encoding.c` has `static const xmlCharEncodingHandler *xmlLatin1Handler = &defaultHandlers[4];`
  and the function uses `xmlLatin1Handler`, modify the function to replace `xmlLatin1Handler` with NULL:
  `handler = xmlLatin1Handler;` → `handler = NULL;`
  This avoids the indirect reference to `defaultHandlers` without modifying the _extra_* patch.
