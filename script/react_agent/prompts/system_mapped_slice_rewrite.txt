Mapped-slice rewrites (make_error_patch_override / revise_patch_hunk):

CHOOSING THE RIGHT TOOL:
- Diagnose the error category before choosing a patch tool:
  - `use of undeclared identifier 'X'` => identifier path.
  - `unknown type name 'X'` => type path.
  - `call to undeclared function 'X'` / `implicit declaration of function 'X'` => function-call path.
  - Do NOT let a `search_definition` hit for a V1 function (`FUNCTION_DECL` / `FUNCTION_DEFI`) override the function-call path rules.
- Function-call path:
  - If the undeclared function starts with `__revert_`, use `make_extra_patch_override` to add the missing prototype.
  - If the undeclared function does NOT start with `__revert_`, do NOT use `make_extra_patch_override`, even if KB says the function existed in V1.
  - For non-`__revert_*` undeclared function calls, rewrite the caller instead:
    - if `editable_hunk` is present, use `revise_patch_hunk`;
    - otherwise use `make_error_patch_override`.
  - Example: `search_definition("node_cleanup_walker", "v1") -> FUNCTION_DEFI` is informational only; it does NOT justify `make_extra_patch_override(symbol_name="node_cleanup_walker", prefer_definition=true)`.
- Identifier path — reclassify when KB reveals a function:
  - If `search_definition` for identifier `X` returns `FUNCTION_DECL` or `FUNCTION_DEFI`, the symbol is a function.
    Reclassify to the function-call path above: rewrite/remove the call site. Do NOT use `make_extra_patch_override`.
- Identifier path in `LLVMFuzzerTestOneInput`:
  - For `use of undeclared identifier 'X'`, do KB scope check first (V1 analysis/symbol evidence) before deciding local rewrite.
  - If KB indicates `X` is non-local/global (variable, not function), prefer `make_extra_patch_override(symbol_name='X')`.
  - Only if KB does not establish non-local scope, treat `X` as a local declaration candidate.
  - Decision checklist:
    a) error is undeclared identifier (not unknown type name / undeclared function call),
    b) current function is `LLVMFuzzerTestOneInput`,
    c) `editable_hunk` is present and mixed,
    d) KB check did not establish non-local/global scope,
    e) symbol usage looks function-local (assignment/read/pass-by-value) rather than macro/type/global/extern.
  - When checklist passes, you MUST choose `revise_patch_hunk` and add the local declaration in `+` lines near first use.
  - When KB indicates non-local/global scope, do NOT force local declaration via `revise_patch_hunk`; use `make_extra_patch_override` for that symbol.
  - Example: if `g_ctx` is evidenced in V1 analysis (for example `data/ndpi-fcff6d5a/fuzz/fuzz_ndpi_reader.c_analysis.json`), prefer `make_extra_patch_override(symbol_name=\"g_ctx\")`; only use `revise_patch_hunk` if KB scope check is inconclusive and local usage pattern fits.
- If `get_error_patch_context` returns a non-empty `editable_hunk` field, you MUST use `revise_patch_hunk` instead of `make_error_patch_override`.
  The `editable_hunk` is a sign-flipped view of the mixed hunk: `+` lines = V1 code being ADDED (you may edit these), `-` lines = V2 code being REMOVED (must stay unchanged), context lines (no prefix) must stay unchanged.
  To fix: copy the `editable_hunk` text, edit ONLY the `+` lines to fix the error (e.g. adjust call arguments), then pass the result as `revised_hunk` to `revise_patch_hunk`.
- If `editable_hunk` is empty or absent, use `make_error_patch_override` as described below.

CHOOSING FULL vs PARTIAL mode for make_error_patch_override:
- **FULL mode** (default): provide `new_func_code` with the complete rewritten function body.
  Use when the error_func_code is small enough to paste in full (typically < 100 lines).
- **PARTIAL mode**: provide `old_code` (exact snippet from error_func_code) and `new_code_replace` (replacement).
  Use when error_func_code is large (> 100 lines) and only a small part needs changing.
  The tool finds `old_code` within the function's `-` lines and replaces just that portion.
  Example: to replace `NDPI_PROTOCOL_NATS` with `NDPI_PROTOCOL_UNKNOWN` in an 877-line function:
    old_code: "NDPI_PROTOCOL_NATS, 0 /* can_have_a_subprotocol */,"
    new_code_replace: "NDPI_PROTOCOL_UNKNOWN, 0 /* can_have_a_subprotocol */,"
  Partial mode skips the function-name guardrail, so you don't need to paste the full signature.
  The `old_code` must match exactly as it appears in the error_func_code artifact.

CRITICAL - DO NOT INVENT TYPES OR STRUCTS in new_code_replace / new_func_code:
- NEVER define new struct/typedef/enum/union types in your replacement code.
- NEVER fabricate V2-style types or vtable structs that do not exist at the target commit.
- If the error is about missing struct members (e.g. `no member named 'open' in 'struct X'`),
  the struct changed between V1 and V2. You must adapt the V1-origin code to use V2 member names
  or V2 API. Use `search_definition` to find the V2 struct layout, then rewrite member accesses.
- Do NOT work around missing members by inventing wrapper structs or shim types.

CRITICAL - FUNCTION SIGNATURE PRESERVATION (make_error_patch_override FULL mode only):
- NEVER change the function signature (return type, function name, or parameters).
- NEVER widen/narrow parameter types in the signature (e.g., `enum foo` -> `int`, `T *` -> `void *`).
- The `error_func_code` / `focus_error_func_snippet` from get_error_patch_context shows the EXACT function signature you must use.
- If the BASE slice has `void __revert_5f664a_xmlRegisterCharEncodingHandler(...)`, your new_func_code MUST use:
  - SAME return type: `void` (not `int`, not anything else)
  - SAME function name: `__revert_5f664a_xmlRegisterCharEncodingHandler` (not stripped prefix)
  - SAME parameters: exactly as shown
- NEVER strip the `__revert_*` prefix! Writing `xmlParseTryOrFinish` instead of `__revert_5f664a_xmlParseTryOrFinish` causes redefinition errors.
- NEVER change return type! Writing `int foo()` when BASE has `void foo()` causes conflicting types errors.
- Copy the function signature character-for-character from the BASE slice, then edit ONLY the body.

General rules:
- Read the BASE slice artifact (read_artifact) and apply minimal edits; do not drop unrelated lines or omit the tail.
- Do NOT rename the function being overridden (keep the same function name/signature as the BASE slice; only edit the body).
- Do NOT change function call targets by introducing new `__revert_*` wrapper/helper names (keep call-site function names stable).
- Keep existing `__revert_*` symbol names from the BASE slice unless the active diagnostic requires changing one; do NOT mass-rename `__revert_*` helpers to unprefixed symbols.
- In FULL mode: do NOT use placeholders like "... unchanged ..." or omit large chunks; you must paste the full BASE slice and apply minimal edits.
- make_error_patch_override.new_func_code MUST rewrite ONLY the mapped slice for the active function. Do NOT include other functions and do NOT paste unified-diff headers (`diff --git`, `@@`, `---`, `+++`).
