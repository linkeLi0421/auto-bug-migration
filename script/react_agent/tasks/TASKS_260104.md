## Next: Enforce patch generation as the last tool step

- calls `read_artifact` too early (pulls large diffs/function bodies into the prompt before analysis),
- calls `make_error_function_patch` before checking the relevant V1/V2 definitions (e.g. struct member drift),
- then continues calling analysis tools after already generating a patch.

This produces confusing output and can generate incorrect patches.

### Goal

Adopt a strict “two-phase” workflow:

1) **Analysis phase** (no `read_artifact`, no patch generation)
   - map error -> patch context

   - fetch/compare relevant V1 vs V2 definitions
   - decide the V2-semantic adaptation strategy

2) **Patching phase** (right before the end)
   - call `read_artifact` only to fetch the minimum needed code context
   - call `make_error_function_patch` as the **last tool call**

## Next: Fix `make_error_function_patch` (patch-bundle rewrite)

However, `make_error_function_patch` currently generates a **standalone** unified diff against the source file. In practice, we want the reverse operation of `get_error_v1_function_code`:

- take a replacement function body (from the agent),
- and **rewrite the patch bundle’s function slice** by replacing the existing `-` lines with the new function body (each line stored as `-<line>`).

This keeps the patch bundle as the single source of truth for the recreated function code.

### Goal

Given `(patch_path, file_path, line_number, new_func_code)`:

- map to `(patch_key, func_start_index, func_end_index)` using the same mapping as `get_error_v1_function_code`,
- replace only the `-` lines within that slice with `-` + the provided function lines,
- keep non-`-` lines in the slice untouched (context / `+` lines, if any),
- recompute and update the `@@ -a,b +c,d @@` hunk lengths so the resulting diff stays valid,
- return the updated `patch_text` (artifact-backed) plus a small summary (`patch_key`, `old_signature`, line counts).

## Next: Fix `make_error_function_patch` hunk headers (start lines + lengths)

### Problem

`make_error_function_patch` rewrites a patch-bundle slice and updates the `@@ -a,b +c,d @@` *lengths*, but it keeps the
`old_start/new_start` line numbers unchanged.

When the replacement changes the per-hunk line delta (`new_len - old_len`), any later hunks in the same diff need their
`+new_start` adjusted; otherwise the unified diff is internally inconsistent and downstream tools can mis-map line numbers.

Example artifact:

- `data/react_agent_artifacts/20260105_002454_895631_e2c75a27/make_error_function_patch_patch_text_error.c.diff`
- `@@ -637,229 +637,6 @@` should be recomputed when the hunk shape changes.

### Goal

After `make_error_function_patch` rewrites the patch-bundle text, the unified diff should be self-consistent:

- each hunk header has correct `old_len/new_len`
- each hunk header has correct `new_start` (shifted by cumulative deltas from previous hunks)
- `old_start` remains stable (relative to the pre-patch file)

### Plan / Tasks

- [x] Update `script/migration_tools/tools.py` (`make_error_function_patch`) to recompute hunk headers fully:
  - parse hunks
  - recompute per-hunk `old_len/new_len` from body lines
  - recompute `new_start` for each subsequent hunk using cumulative delta (`Δ += new_len - old_len`)
- [x] Add a regression fixture with 2+ hunks where replacing the function slice changes the hunk delta; assert the later hunk’s `+new_start` changes accordingly.
- [x] Run `bash script/migration_tools/test_migration_tools.sh` and `bash script/react_agent/test_langgraph_agent.sh`.
