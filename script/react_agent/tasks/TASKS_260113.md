# TASKS

## Patch-apply failures must not be treated as “fixed”

### Context / Repro

- In some OSS-Fuzz runs the generated override diff is invalid, so `git apply` fails with e.g.:
  - `error: corrupt patch at line 402`
  - Example log: `data/react_agent_artifacts/tail-parser.c-xmlParse3986DecOctet_/ossfuzz_apply_patch_and_test_build_output.6.log`
- Today the agent often decides “target error fixed” just because the target compiler errors from that hunk are no longer present in the build log.
- When the patch fails to apply, the build may succeed on the unmodified source tree, so the target errors “disappear” even though nothing was fixed.

### Root cause (current behavior)

- `script/fuzz_helper.py` uses `git apply ... --reverse /patch;` without `set -e` / return-code checking, so patch-apply failures don’t fail `build_version`.
- `script/react_agent/agent_langgraph.py:_summarize_target_error_status` only parses `file:line:col: error:` style compiler errors (`script/react_agent/build_log.py:iter_compiler_errors`) and ignores patch-apply failures like `error: corrupt patch at line N`.

## Plan

- [x] Confirm failure mode in `ossfuzz_apply_patch_and_test_build_output.6.log` and locate where it’s missed (verdict logic + fuzz_helper patch apply).
- [x] Add a dedicated “patch apply failure” detector (regex) for OSS-Fuzz logs:
  - `error: corrupt patch at line`
  - `error: patch failed:`
  - `error: .*: patch does not apply`
  - `patch: ****` (malformed patch)
  - Keep it separate from compiler error parsing (these lines have no `file:line:col`).
- [x] Wire patch-apply failure detection into OSS-Fuzz verdicts:
  - In `script/react_agent/agent_langgraph.py:_summarize_target_error_status`, if patch-apply failure is seen in build log, return `status=failed` with `reason` + `hint`, and never claim `fixed=yes`.
  - Do the same for patch-scope reporting (`_summarize_active_patch_key_status`) so multi-agent summaries clearly show “patch did not apply”.
- [x] Make `build_version` fail fast when patch application fails:
  - In `script/fuzz_helper.py:build_version`, check `git apply` return code (and run `git apply --check`) so patch failures propagate as non-zero exit.
- [x] Belt-and-suspenders in the tool layer:
  - In `script/react_agent/tools/ossfuzz_tools.py:ossfuzz_apply_patch_and_test`, scan `build_output` text for patch-apply failures and expose `patch_apply_ok` / `patch_apply_error` in the tool output (even if `returncode==0` due to legacy behavior).
- [x] Update the agent loop behavior on patch-apply failure:
  - Treat as a hard failure for the current override diff (don’t advance to “next error”).
  - Next step guidance should explicitly say: “override diff is corrupt / doesn’t apply; regenerate `make_error_patch_override` output (ensure unified-diff header + valid hunk).”
- [x] Add regression tests:
  - In `script/react_agent/test_langgraph_agent.sh`, feed a fake build log artifact containing `error: corrupt patch at line 402` and assert verdict is `status=failed` (not `fixed=yes`).
  - Ensure final output includes the reason and the log artifact path.

## Better merged/tail hunk support: make `get_error_patch_context` self-sufficient

### Idea (yes, good idea)

Consolidate “what code do we edit?” into a single tool call:

- `get_error_patch_context` already maps build error → `(patch_key, func_start_index, func_end_index, old_signature)` and writes the full hunk excerpt.
- For merged/tail hunks, the agent needs two separate “views”:
  1) the full hunk (for patch validity + context), and
  2) the specific function slice the build-error line points to (to keep the LLM focused).
- If `get_error_patch_context` also returns/stores both the full `-` code (entire hunk) and the per-function `-` code (mapped slice), we can drop the extra `get_error_v1_code_slice` tool call in the graph.

### Plan

- [x] Define the new `get_error_patch_context` output schema:
  - Keep existing `excerpt` artifact (full unified-diff, applyable).
  - Add `patch_minus_code` (all `-` lines in the hunk; “pure patch text”).
  - Add `error_func_code` (only `-` lines within the mapped `(func_start_index, func_end_index)` slice; function containing the error line).
  - Add line-count metadata for both (`*_lines_total`, `*_truncated` should be false / removed).
- [x] Implement extraction in `script/migration_tools/tools.py:get_error_patch_context`:
  - Reuse the same hunk/body indexing logic as `make_error_patch_override` (body_start + func_start/end).
  - For `patch_minus_code`, reuse `_extract_minus_lines_from_unified_diff(excerpt)`.
  - For `error_func_code`, slice the hunk body by indices and collect only `-` lines.
  - Handle edge cases: missing indices / no `-` lines → return empty string + clear `note`.
- [x] Update artifact offloading so the new fields are persisted automatically:
  - Extend `script/react_agent/artifacts.py:_PATCH_TOOL_FIELDS["get_error_patch_context"]` to offload `patch_minus_code` and `error_func_code` (e.g. `.c`).
  - Provide small “focus” snippets (like `focus_excerpt`) for at least `error_func_code` so the model can see a preview without extra reads.
- [x] Make the agent stop calling `get_error_v1_code_slice`:
  - Update `script/react_agent/agent_langgraph.py` initial patch-scope flow to rely on `get_error_patch_context.error_func_code`/`patch_minus_code`.
  - Keep `get_error_v1_code_slice` temporarily as a compatibility wrapper (optional), but remove it from the required/default tool chain.
- [x] Update/adjust tests:
  - `script/react_agent/test_langgraph_agent.sh`: update assertions that currently expect `get_error_v1_code_slice` early in the tool sequence.
  - Add a merged/tail fixture test: ensure `error_func_code` is a *subset* of `patch_minus_code` and corresponds to `old_signature`.
- [x] Deprecate `get_error_v1_code_slice` in docs:
  - Marked deprecated in `script/react_agent/tools/registry.py`.
- [x] (Follow-up) Remove `get_error_v1_code_slice`:
  - Removed from tool registry/runner and deleted the underlying implementation; updated tests/docs accordingly.

## Don’t use `search_definition` to “find struct fields”

### Context

In missing-struct-member errors (e.g. `no member named 'nsdb' in 'struct _xmlParserCtxt'`), the agent sometimes calls
`search_definition(symbol_name=<something>, ...)` trying to “search a field”. `search_definition` is a *symbol* definition
lookup and is not a reliable way to locate struct *members* (fields are not standalone symbols in the same way; results
are often empty or misleading).

### Plan

- [x] Clarify tool semantics:
  - Update tool descriptions + the system prompt to state: `search_definition` is for top-level symbols (struct/typedef/function),
    not struct fields; for fields use `kb_search_symbols(..., kinds=['FIELD_DECL', ...])` and/or read the parent struct body.
- [x] Add a guardrail: if the model tries `search_definition` on a missing member name (or `ctxt->member` / `ctxt.member`),
  rewrite to `kb_search_symbols` for that identifier in `v2` (kinds include `FIELD_DECL`, `VAR_DECL`, `MACRO_DEFINITION`).
- [x] Add a regression test in `script/react_agent/test_langgraph_agent.sh` to lock the rewrite behavior.
- [x] Run `bash script/react_agent/test_langgraph_agent.sh`.

## “Target error fixed” should be per active function group (merged hunks)

### Context

For merged/tail hunks we now iterate one `old_signature` (function) at a time. Multiple functions can fail with the same
compiler message (e.g. `no member named 'nsdb'...`), so the current verdict logic may report:

`Target error fixed: no`

even when *the active function’s* errors are gone (because the same message still exists in other functions in the same
patch_key).

### Plan

- [x] Change the target-error verdict to be function-aware in patch-scope runs:
  - If `state.active_old_signature` is set, treat “target fixed” as: **no remaining compiler errors mapped to**
    `(active_patch_key, active_old_signature)` in the latest OSS-Fuzz build logs.
- [x] Keep existing behavior for non patch-scope runs (or when `active_old_signature` is empty).
- [x] Add a regression test that reproduces the false-negative case: same msg in same patch_key but different
  `old_signature` should still yield `fixed=yes` for the active signature.
