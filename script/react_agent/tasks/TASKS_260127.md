## 2026-01-28: Fix `_extra_*` override merge reordering bug (COMPLETED)

**Problem:** When multiple override diffs exist for the same `_extra_*` key (even if identical), the `_merge_extra_hunk_override_texts` function reordered blocks by category, causing `typedef` declarations (category 3) to end up AFTER function prototypes (category 2). This broke compilation when prototypes referenced the typedef (e.g., `xmlParserNsData *__revert_e11519_xmlParserNsCreate(void);`).

**Root Cause:** In `_extra_merge_block_category()`:
- `typedef` and `tag` blocks were not handled explicitly, falling through to category 3
- `prefer_prepend=True` override for typedef/tag gave category 0, putting them BEFORE macros (also wrong)

**Solution:**
1. Updated `_extra_merge_block_category()` to check `define` and `typedef/tag` BEFORE `prefer_prepend`, ensuring correct ordering: define(1) < typedef/tag(2) < prototype(3)
2. Added optimization to skip merge when all override texts are identical (preserves agent's intended ordering)
3. Added regression tests verifying typedef appears before prototypes that reference it

**Changes Made:**
- [x] Fixed `_extra_merge_block_category()` in `script/react_agent/tools/ossfuzz_tools.py`
- [x] Added identical-text detection in `_merge_extra_hunk_override_texts()`
- [x] Added regression tests in `script/react_agent/test_langgraph_agent.sh`

**Testing:** All tests in `test_langgraph_agent.sh` pass.

## 2026-01-28: make_link_error_patch_override State Handler Missing (COMPLETED)

**Problem:** When `make_link_error_patch_override` is called and succeeds, the generated override diff is NOT added to `st.patch_override_paths`. When `ossfuzz_apply_patch_and_test` is subsequently called, it uses empty `patch_override_paths` instead of including the newly generated override diff.

**Root Cause:** In `agent_langgraph.py`, there are post-observation handlers for:
- `make_error_patch_override` (updates `st.patch_override_paths`)
- `make_extra_patch_override` (updates `st.patch_override_paths`)

But there was NO handler for `make_link_error_patch_override`. The tool exists in the registry, is implemented, and executed, but the agent never processed its output to update state.

**Solution:** Added a handler block for `make_link_error_patch_override` in `agent_langgraph.py` similar to the existing `make_error_patch_override` handler.

**Changes Made:**
- [x] Added handler for `obs.tool == "make_link_error_patch_override"` after the `make_extra_patch_override` handler (~lines 5715-5793)
- [x] Extract `patch_text` from the tool output and update `st.patch_override_by_key` and `st.patch_override_paths`
- [x] Call `_write_effective_patch_bundle()` to update `st.patch_path` with the merged bundle
- [x] Added debug logging similar to other override handlers

**Testing:** All tests in `test_langgraph_agent.sh` pass.

## 2026-01-28: Linker Error Strategy Prompt (COMPLETED)

**Problem:** Agent was trying to implement missing symbols for linker errors instead of removing/guarding the code that references them.

**Solution:** Created `system_linker_error.txt` prompt fragment with clear guidance to:
- NOT implement missing symbols
- REMOVE or GUARD the specific code that references the missing symbol
- Only modify the function with the linker error, not other functions
- Use minimal changes (remove calls, use `#if 0`, replace with defaults)

- [x] Created `prompts/system_linker_error.txt` with linker error handling strategy

## 2026-01-28: Linker Error Detection in OSS-Fuzz Verdict (COMPLETED)

**Problem:** After agent generates a patch and runs ossfuzz_apply_patch_and_test, the verdict logic (`_summarize_active_patch_key_status`) only checks for compiler errors, not linker errors. If linker errors remain (like `undefined reference to '__revert_e11519_xmlGlobalInitMutexUnlock'`), the agent reports "Active function fixed: unknown" instead of detecting the remaining error.

**Root Cause:** `_iter_ossfuzz_compiler_errors` only called `iter_compiler_errors`, not `iter_linker_errors`. Additionally, the status function skipped errors with `line <= 0` (which linker errors have).

**Changes Made:**
