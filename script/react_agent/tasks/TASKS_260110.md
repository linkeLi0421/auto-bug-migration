# React agent: loop on remaining errors in the same hunk

## Goal
When `script/react_agent/agent_langgraph.py` is running in `--error-scope patch` mode and using OSS-Fuzz:

- After each `ossfuzz_apply_patch_and_test` tool call, re-parse the new build output.
- If the just-targeted error is fixed but there are still compiler errors mapping to the same patch **hunk/slice**, keep iterating (read current BASE slice → generate next override → re-test) instead of drifting to unrelated actions.
- Each iteration must start from:
  - the original patch bundle (`patch_path`) and
  - the latest `make_error_patch_override` `patch_text` from the previous loop (so mapping/slices reflect the current state of the hunk).

## Plan
### 1) Reproduce + pinpoint the break
- Use `log/agent_log/tmp1.log` to identify the exact transition where the agent stops progressing to the next error in the same patch slice after the first OSS-Fuzz run.
- Confirm which state fields are stale after the run (`grouped_errors`, `target_errors`, `patch_key`, macro-guardrail fields, and any cached patch mapping).

### 2) Track “active patch slice” explicitly
- Extend `AgentState` to persist the *current* patch-scope focus across iterations:
  - `active_patch_key` (string)
  - `active_file_path` (string)
  - `active_line_number` (int, from the selected error)
  - `active_func_start_index` / `active_func_end_index` (slice bounds in the hunk; from `_get_error_patch_from_bundle` / `get_error_patch_context`)
  - `active_excerpt_artifact_path` (optional, for context/debug)
- Clear/refresh macro guardrail state when the active error/slice changes (`macro_tokens_not_defined_in_slice`, `macro_lookup`) so it can’t force unrelated macro work in the next loop.

### 3) Build an “effective” patch-bundle view (bundle + last override patch_text)
- Implement a helper in `script/react_agent/agent_langgraph.py` to load the patch bundle and overlay the latest override patch text for `active_patch_key`:
  - Read the last `make_error_patch_override` `patch_text` artifact (from `state.patch_override_paths[-1]` or `state.patch_result.patch_text.artifact_path`).
  - Replace the in-memory `PatchInfo.patch_text` for that key with the override text.
  - Recompute the patch’s derived line ranges (`old_start_line/end_line`, `new_start_line/end_line`) from the override’s hunk headers.
  - Rebuild `PatchBundle.keys_sorted` (and any indexes used by mapping) so `_get_error_patch_from_bundle` sees correct offsets after the override.
- Use this “effective bundle” for any post-OSS-Fuzz error→patch-key mapping and for selecting the next slice bounds.

### 4) Add a post-OSS-Fuzz state-graph loop (conditional edge)
- In `_run_langgraph`, add a new deterministic node, e.g. `post_ossfuzz_refresh`.
- Add a conditional edge out of the `tool` node:
  - If `state.last_observation.tool == "ossfuzz_apply_patch_and_test"` and the run produced compiler errors, route to `post_ossfuzz_refresh`.
  - Otherwise route back to `llm`.
- `post_ossfuzz_refresh` responsibilities:
  - Parse `build_output` (and `check_build_output` if present) from the OSS-Fuzz tool output.
  - Map parsed errors to patch keys using the “effective bundle”.
  - If there are still errors mapping to `active_patch_key` **and** inside the same slice/hunk, update:
    - `state.grouped_errors`, `state.error_line`, `state.snippet`
    - `state.target_errors` (from the refreshed grouped errors)
    - `state.auto_iterate_pending = True`
    - reset `state.patch_generated`, `state.patch_result`, `state.ossfuzz_test_attempted`
  - If no remaining errors in that slice/hunk (or run budget exhausted), set `auto_iterate_pending = False` and allow the normal “final” path.

### 5) Tighten the iteration criterion (“fixed one error” → keep going if others remain)
- Update the post-OSS-Fuzz refresh logic to continue even when the *previous* target error is fixed, as long as:
  - new compiler errors remain in the same slice/hunk, and
  - `ossfuzz_runs_attempted < max_ossfuzz_runs`.
- Ensure we always drive the next cycle from:
  - the patch bundle + last override `patch_text` (for mapping/slice bounds), and
  - the current BASE slice artifact (for generating `new_func_code`).

### 6) Regression tests
- Add a regression test (extend `script/react_agent/test_langgraph_agent.sh` or add a small Python test) that simulates:
  - an initial patch-scope error list with multiple errors in one patch slice,
  - a first OSS-Fuzz result where the first error disappears but another in-slice error remains,
  - expected behavior: the agent refreshes errors and selects the next in-slice error for the next override (no unrelated macro/tool detours).

## Done criteria
- After the first `ossfuzz_apply_patch_and_test`, the next step is deterministically:
  `parse new build output → pick next remaining error in the same slice → read BASE slice → make_error_patch_override → ossfuzz_apply_patch_and_test`.
- The agent stays in the loop until that slice/hunk is clean (or `--max-ossfuzz-runs` is reached).
