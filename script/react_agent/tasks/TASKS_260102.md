## Patch-aware triage for migrated build logs

### Problem

When the build log comes from **post-migration / post-patch** code, the reported `file:line:col` often does **not** match the line numbers in the raw V1/V2 source checkouts.

Current behavior observed: for `unknown type name ...` the agent may call `read_file_context(file_path, line_number, version=v2)` using the build-log location, which can be misleading because that location is in the migrated code.

### Goal

For “migrated-code” errors, prefer a **patch-first workflow**:

1) Parse the build error (`parse_build_errors`)
2) Map the error location to a patch hunk (`get_error_patch`)
3) Inspect the patch diff for the relevant change (`get_patch`)
4) Then look up missing types/macros/decls in V1 (and/or V2) (`search_definition`, `inspect_symbol`)

### Tasks

1) Make the patch bundle available to the agent
- [x] Add `--patch-path` CLI arg (and env var `REACT_AGENT_PATCH_PATH`) to `script/react_agent/agent_langgraph.py`
- [x] Optionally infer a default `patch_path` when not provided (e.g. from `--v2-src` repo name → `data/tmp_patch/<name>.patch2`)
- [x] Include `patch_path` in the initial prompt so the model can call patch tools without guessing paths

2) Teach the agent “migrated line numbers” vs “source line numbers”
- [x] Add a prompt rule: when `patch_path` is present, treat build-log `file:line` as **migrated code** and avoid `read_file_context` unless explicitly requested by the user
- [x] For `unknown_type_name` (and other undeclared identifiers/types), prefer the first step to be `parse_build_errors`, then patch mapping (`get_error_patch_context` / `get_error_patch`)

3) Reduce steps with a higher-level patch-context tool (optional, but recommended)
- [x] Add `get_error_patch_context({patch_path, file_path, line_number, context_lines?})` returning:
  - patch key + old signature
  - a small diff excerpt around the mapped hunk/offset
  - the extracted “unknown symbol” name(s) if present in the error message
- [x] Keep output bounded (line limits, omit full patch text by default)

4) Improve lookup when the “unknown type” is actually a macro
- [x] Add a read-only source grep tool limited to repo roots (e.g. `search_text({query, version, limit, file_glob?})`) for cases where `search_definition` cannot find macros in the clang JSON
- [x] Update the prompt to try `search_definition` first, then fallback to grep (`#define`, `typedef`, `struct`) for macro-like tokens

5) Tests
- [x] Add a fixture build log containing `unknown type name 'FOO'` and verify the stub model path prefers patch mapping when `--patch-path` is provided
- [x] Extend `script/react_agent/test_langgraph_agent.sh` (or add a new test script) to assert:
  - `parse_build_errors` is called before `get_error_patch`
  - `read_file_context` is not called for migrated-line-number cases

### Success criteria

- [x] With `--patch-path data/tmp_patch/<target>.patch2`, the agent uses patch tools first for `unknown type name` errors (no `read_file_context` by default)
- [x] Agent output includes: (a) patch key, (b) old signature/function context, (c) suggested next inspection step (e.g. missing include or typedef location in V1)

## Next: Handle multiple errors per patch

### Problem

Right now `script/react_agent/build_log.py` finds only the **first** compiler error (`find_first_fatal`) and the agent plans around that single line. In real migrated builds, many errors can share the same root cause and often map to the **same patch**.

### Goal

When `--patch-path` is available, allow the agent to triage **all errors that map to the same patch key** in one run:

- collect N compiler errors from the build log
- map each error location → patch key via `get_error_patch`
- pick a target patch group (e.g. same patch as the first error, or the patch with most errors)
- give the model a compact list of all errors in that patch + patch context

### Tasks

- [x] Add `iter_compiler_errors(build_log, snippet_lines=...) -> list[{file,line,col,msg,raw}]` to `script/react_agent/build_log.py`

2) Add patch-scoped triage mode to the agent

- [x] If `--error-scope patch` and `--patch-path` is set:
  - map errors to `patch_key` using `get_error_patch`

3) Provide patch context for the whole group

## Next: Improve `search_definition` for typedef/type_ref cases

### Problem

`search_definition(symbol_name, version)` currently selects a single “best” node and prints the code for that node’s `extent`.

For types, this often returns a **forward typedef** (e.g. `typedef struct _T T;`) or **forward decl**, even when the KB includes richer nodes like `TYPE_REF` that point to the **real definition** via:

- `type_ref.underlying.extent`
- `type_ref.typedef_extent`

This makes the agent fall back to `search_text` (grep) and/or manual source browsing, even though the KB already contains the definition location.

### Goal

Make `search_definition` return a bounded “definition bundle”:

- the current best match (as today)
- plus related/underlying definition snippets when available (typedef target, struct/enum/union definition)
- keep output human-friendly and bounded (avoid dumping huge headers)

### Tasks

1) Add KB helpers to expand “related definitions”
- [x] In `script/react_agent/core/kb_index.py`, add a helper that can derive related definition candidates from a node:
  - from `TYPE_REF.type_ref.usr` (query by USR)
  - from `TYPE_REF.type_ref.underlying.{decl_location,extent}`
  - from `TYPEDEF_DECL.typedef` (follow typedef chains like `T*` → `T` → `struct _T`)

2) Enhance `search_definition` output format
- [x] In `script/react_agent/tools/symbol_tools.py`, update `search_definition` to return multiple sections:
  - `Primary match` (current behavior)
  - `Related definitions` (0..N snippets, each with kind + file:line + code)
- [x] Add strict bounds: `max_snippets` (e.g. 3) and `max_lines_per_snippet` (e.g. 120), with truncation markers.

3) Prefer “real” definitions over forward typedefs/decls
- [x] If the primary match is a 1-line typedef/forward decl and a related candidate has a larger extent (multi-line struct/enum body), surface that candidate as “Real definition”.

4) Tests
- [x] Add a small fixture KB + source tree where:
  - `TYPEDEF_DECL` is a 1-liner forward typedef in one header
  - a corresponding `TYPE_REF` points to an `underlying.extent` in another header containing the real struct body
- [x] Add a test that asserts `search_definition("T", version="v2")` includes both the typedef and the underlying struct definition path/lines (and stays bounded).

### Success criteria

- [x] For a typedef-to-struct case (e.g. `xmlParserCtxt`), `search_definition(..., version="v2")` returns both the alias typedef and the underlying struct body (without needing `search_text`).
- [x] Output remains readable in `--output-format text` and bounded for large types.
